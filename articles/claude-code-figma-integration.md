## 何が起きたのか

2026年2月17日、Figmaが[Claude Code統合を正式発表](https://www.figma.com/blog/introducing-claude-code-to-figma/)しました。これは単なる連携機能の追加ではありません。**コードで作ったUIをブラウザから直接キャプチャして、Figmaキャンバス上で編集可能なフレームに変換する**という、デザインとコードの境界を根本から問い直す動きです。

さらに、[Figma MCP server](https://www.figma.com/blog/introducing-figma-mcp-server/)との組み合わせにより、**Figma → Code** の逆方向の流れも実現。つまり、**Code ⇄ Design の完全な往復**が可能になったということです。

この記事では、何が変わり、どう使えて、開発者とデザイナーそれぞれにどんな影響があるのかを整理します。

---

## 従来の課題：コードとデザインの分断

### コードの強み — 収束のスピード

Claude Codeのようなツールを使えば、プロンプトから数分でプロトタイプが動きます。リアルなデータ、実際のインタラクション、技術的な制約も見えます。**1人で高速に収束する**には最適です。

しかし、UIが複数画面に広がった瞬間、問題が浮上します：

- スクリーンショットを撮って共有 → 静的で編集できない
- 動画を録画 → 重くて探しにくい
- ローカルでビルドを起動してもらう → 環境構築の手間
- 「あの画面、もう一回見せて」→ context switchのコスト

**コードは線形です**。1画面ずつリフレッシュして確認する。でもチームで議論するには、**全体を一度に見渡せる2次元の空間**が必要でした。

### デザインツールの強み — 発散の自由

Figmaのようなデザインツールは、**並べる・複製する・比較する・分岐する**のが得意です。複数のアイデアを同時に並べて「どっちがいい?」と議論できる。アノテーションも自由。

でも、コードで書いたUIを取り込むには、これまで手動で再現するしかありませんでした。**それは単なる手間じゃなく、情報のロス**でもあります。実装の詳細、インタラクション、技術的な制約が、デザインに反映されない。

## Claude Code → Figma：実装をキャンバスへ

### 仕組み

Claude Codeで作ったUI（本番・ステージング・localhost問わず）をブラウザで開き、キャプチャ機能を使うと：

1. **そのまま画像として**クリップボードにコピーされる
2. Figmaに貼り付けると**編集可能なフレーム**に変換される
3. 複数画面をキャプチャすれば**フロー全体**が保存できる

重要なのは、**スクリーンショットではなく、編集可能なFigmaフレーム**だということ。つまり：

- 要素を動かせる
- テキストを書き換えられる
- 色やレイアウトを調整できる
- アノテーションを追加できる
- 複製して別案を作れる

### 何が変わるか

#### 1. システム全体を俯瞰できる

複数画面を並べて、フロー全体の矛盾や不整合を視覚的に発見できます。「この2画面、ヘッダーのボタンが違う」のような問題は、並べなければ気づきません。

#### 2. 実装せずに代替案を試せる

フレームを複製して、「このボタンを右に寄せたら?」「この画面をスキップしたら?」を試せます。コードを書き直さずに、構造的な変更をシミュレートできる。

#### 3. 意思決定が早くなる

デザイナー・エンジニア・PMが**同じアーティファクトを見ながら**議論できます。「この実装で合ってる?」「ここ、もう少し余白ほしい」が、全員同じ粒度で話せる。

#### 4. 実装が共有資産になる

「あのプロトタイプ、どこ?」がなくなります。Figmaに貼れば、リンク1本で共有でき、履歴も残ります。捨てられたアイデアも、後から掘り起こせる。

## Figma MCP → Code：デザインを実装へ

逆方向も整備されています。[Figma MCP server](https://www.figma.com/blog/introducing-figma-mcp-server/)を使えば、FigmaフレームをClaude Codeに渡して、**デザインに忠実なコードを生成**できます。

### ワークフロー例

1. Claude Codeでプロトタイプを作る
2. Figmaにキャプチャして、チームでレビュー・改善
3. 改善後のFigmaフレームをClaude Codeに戻す
4. MCP経由でコードを更新

これは**片道切符じゃない**ということです。「コードで試す→デザインで広げる→コードに戻す」のループが、摩擦なく回せる。

---

## 実際のワークフロー：こう使う

### パターン1：エンジニアが先に動く

1. Claude Codeでアイデアをプロトタイプ
2. 主要な3〜5画面をキャプチャしてFigmaに貼る
3. デザイナーに「このフロー、どう思う?」と共有
4. Figma上で並べて議論 → アノテーション追加
5. 合意した方向で実装を進める

**メリット**: エンジニアが「動くもの」で会話をスタートでき、デザイナーは実装の制約を見た上で提案できる。

### パターン2：デザイナーが引き取る

1. エンジニアが実装した画面をFigmaに取り込む
2. デザイナーが複製して3パターン作る（色・レイアウト・情報設計を変えて）
3. チームで比較 → 1つに決定
4. 選んだパターンをMCP経由でコードに反映

**メリット**: 実装を壊さずに、構造レベルの探索ができる。

### パターン3：リアルタイム同期

1. Claude Codeでローカル開発中
2. 重要な状態（ログイン前後・エラー状態など）をその場でキャプチャ
3. Figmaの同じファイルに追加していく
4. レビュー時には、すでに全画面がFigma上に揃っている

**メリット**: 「見せるための準備」が要らない。開発しながら自然にドキュメント化できる。

## 開発者への影響：コードはもう孤独じゃない

### 変わること

- **「動くもの」が共有資産になる** — スクショや動画じゃなく、編集可能な形で
- **レビューのハードルが下がる** — 「ちょっと見て」がFigmaリンク1本で済む
- **無駄な再実装が減る** — 「試しに作ったけど没」がFigma上で形を変えて残る

### 変わらないこと

- コードを書くこと自体の価値 — むしろ、**コードで考える**ことの価値が可視化される
- ローカルでの高速なイテレーション — Figmaに移すのは、チームに広げたい時だけ

---

## デザイナーへの影響：実装が素材になる

### 変わること

- **実装済みのUIが編集対象になる** — 再現する手間がなくなる
- **技術的制約が最初から見える** — 「これ実装できる?」の答えが、すでに目の前にある
- **探索の起点が増える** — Figma Makeだけじゃなく、Claude Codeで作られたものも素材に

### 変わらないこと

- デザインする行為そのもの — 並べる・比較する・形にする、は依然としてデザイナーの仕事
- Figmaの使い方 — 新しいツールを覚える必要はない。「貼り付ける」だけ

## これは何の始まりか

### AIがデザインとコードを統合する

Figma MakeでもClaude Codeでも、「プロンプトから形が生まれる」のは同じです。違いは**出力形式**だけ。でも、これまではその違いが大きな壁でした。

この統合が意味するのは、**AIはどちらから始めてもいい**ということ。そして、**どちらの形でも、すぐに反対側に持っていける**ということ。

### 「最初の案」じゃなく「最良の案」へ

AIで生成が簡単になった今、課題は**どれを選ぶか**です。10個のアイデアを1秒で作れるなら、それを並べて比較する場が必要です。

Claude Code → Figmaの統合は、その場を作る動きです。「コードで速く作る → Figmaで広く見る → またコードに戻す」のループが、ツールの壁で途切れなくなる。

---

## まとめ：境界が溶けた先に

かつて「デザイナー」と「エンジニア」は、別々のツールで、別々のフォーマットで、別々のタイミングで働いていました。

Claude CodeとFigmaの統合は、**その境界を溶かします**。コードはデザインになり、デザインはコードになる。どちらから始めても、どちらに戻っても、摩擦がない。

これはツールの話じゃなく、**働き方の話**です。そして、**製品の話**でもあります。境界がなくなれば、もっと速く、もっと自由に、もっといいものが作れる。

その未来が、今日から始まりました。
